Arquitectura Medallion en Azure SQL

Capa BRONCE: Réplica exacta de archivos Excel. Tablas staging sin transformación (BI_llamadas_inconcert_bronce, Mensajes_bitrix_bronce). Carga vía Python pandas to_sql() o bulk insert.

Capa PLATA: Datos limpios y estandarizados. Stored procedures T-SQL ejecutan: conversión tipos de datos, normalización fechas UTC, eliminación duplicados, validación integridad referencial. Ejemplo: llamadas_plata con fecha_hora DATETIME2, duracion_seg INT, campaña_id FK.

Capa ORO: Star schema para analytics. Fact tables (fact_llamadas, fact_conversaciones) + dimensions (dim_fecha, dim_campana, dim_agente). Vistas materializadas para dashboards frecuentes. Indices clustered en fecha + campaña.

Stack técnico recomendado

ETL: Python 3.11 + pandas + pyodbc + sqlalchemy. Script programado Windows Task Scheduler o Azure Functions (serverless).

Transformaciones: T-SQL stored procedures. Ejemplo: CREATE PROCEDURE sp_limpiar_llamadas ejecuta lógica limpieza + auditoría.

BI: Power BI Desktop desarrollo + Power BI Service publicación. Import mode (no DirectQuery) para S0/S1. Refresh programado 03:00 AM.

Monitoreo: Azure SQL Insights (query performance), alertas email si DTU > 80%, auditoría cambios schemas.

Limitación
Costo fijo mensual: Pagas $15-30/mes aunque solo uses 2 horas/día (proceso nocturno). No puedes "apagar" la BD para ahorrar.

Queries complejos lentos en S0: JOIN 60K×40K toma 45-90 seg en S0. Si requieres dashboards real-time con múltiples usuarios concurrentes, necesitas S2+ ($75/mes).

Lock-in Microsoft: T-SQL tiene extensiones propias (OUTPUT clause, MERGE, window functions sintaxis específica). Migración futura a PostgreSQL/MySQL requiere refactorizar stored procedures.

Google BigQuery

Descripción

Data warehouse serverless de Google Cloud con arquitectura columnar. Modelo de pago por uso (storage + queries procesados). Sin infraestructura que administrar.

Estimación costos con volumetría Callcom

Storage (10 GB proyectado):

  • 1.2 GB actual × $0.02 = $0.024/mes
  • 10 GB futuro × $0.02 = $0.20/mes
  • Crecimiento lineal, sin sorpresas

Queries (50 GB scan/mes estimado):

  • Cruces diarios: 10 GB × 30 días = 300 GB scan
  • Con particionado: 300 GB → 50 GB real
  • 50 GB × $0.00625 = $0.31/mes
  • Total: $0.20 + $0.31 = $0.51/mes

⚠️ RIESGO FASE INICIAL: Sin particionado y buenas prácticas, queries exploratorios escanean tablas completas. Ejemplo: JOIN 7 tablas sin filtros = 10 GB scan × 20 pruebas/día = 200 GB/día = $40/día = $1,200/mes primer mes. Con controles: $5-10/mes estable.

Modelo de costos

Storage: $0.02/GB/mes

Queries: $6.25/TB escaneado

10 GB datos: $0.20/mes storage

50 GB queries/mes: $0.30/mes

Total típico: $5-10/mes

Performance

Cruces 10 GB: <5 segundos

Arquitectura: Columnar distribuida

Escalabilidad: Petabytes sin configuración

Procesamiento paralelo masivo, ideal para queries analíticos complejos.

Ventajas

Costo ultra bajo: 95% más barato que Azure SQL (en analytics read-only)

Performance superior: Queries analíticos 5-10x más rápidos

Zero administración: Sin servidor que mantener

Escalabilidad infinita: Crece automáticamente

⚠️ Limitaciones críticas

Ecosistema: Fuera de Microsoft (conector Power BI menos fluido)

Queries costosos: Sin controles, scan full table puede disparar a $200-500/mes

NO tiene UPDATE: Usa MERGE (INSERT + DELETE) → Ineficiente para workloads transaccionales

Patrón Callcom (2K updates/día): 60K MERGE/mes = $50-100/mes real vs $0.51 teórico

Protecciones durante desarrollo

  • Dataset DEV pequeño: 1 semana datos para pruebas (<$1/mes)
  • Dry-run queries: Validar costo antes de ejecutar
  • Límites presupuesto: Alertas en $50/mes
  • Particionado por fecha: Escanear solo días necesarios
  • Clustering por campaña: Optimizar queries frecuentes
Limitación
Fuera de ecosistema Microsoft. Conector Power BI menos fluido que Azure SQL nativo.

RIESGO CRÍTICO INICIO PROYECTO: Sin controles estrictos, queries exploratorios pueden disparar costo a $200-500/mes en primeros meses (scan full table, joins mal optimizados, queries repetitivos sin particionado). Aunque aprendizaje es rápido, requiere disciplina inmediata en buenas prácticas SQL columnar. En Azure SQL el costo es fijo ($15/mes) independiente de errores.

RIESGO CRÍTICO OPERACIONES MERGE: BigQuery NO tiene UPDATE tradicional. Para actualizar registros usa patrón MERGE (INSERT nuevos + DELETE antiguos). Con patrón de uso Callcom (2,000 registros negociaciones/día que requieren actualizar etapa/responsable Bitrix), esto significa 2K × 30 días = 60K operaciones MERGE/mes. Aunque BigQuery cobra ~$0.10 por 1M operaciones DML, en práctica con re-procesos, pruebas y errores de desarrollo el costo real escala a $50-100/mes (vs $0.51 teórico analytics puro). Para workloads transaccionales write-heavy (inserts frecuentes + updates batch), bases relacionales tradicionales (Azure SQL/MySQL) son 10x más eficientes en costo.

✓ CONCLUSIÓN PARA TU CASO DE USO:

Azure SQL S0 (desarrollo) → MySQL On-Premise (producción) es la mejor opción porque:

✗ Opciones descartadas (considerando patrón de uso real)

BigQuery: Aunque es rápido en JOINs analíticos, tu caso requiere INSERT diario (2K registros) + UPDATE masivo Bitrix. BigQuery no tiene UPDATE tradicional, usa patrón MERGE (INSERT + DELETE). Con 2K updates/día × 30 días = 60K operaciones MERGE/mes. Costo: ~$0.10 por 1M operaciones DML, pero con pruebas y re-procesos = $50-100/mes real vs $0.51 teórico. Riesgo crítico fase inicial: Sin controles, queries exploratorios pueden disparar a $200-500/mes. Azure SQL costo fijo $15/mes elimina este riesgo completamente.

Solo Azure SQL permanente: Funciona perfecto técnicamente (mejor para tu patrón INSERT/UPDATE que BigQuery), pero costo $630-900/24 meses vs $0 MySQL on-premise en producción. En desarrollo Azure SQL correcto, en producción migrar a MySQL ahorra $630.

SQL Server on-premise: Licencia $1K-3K/año vs MySQL gratis. Performance idéntico para tu volumetría (2K inserts + 60K×40K join + updates). No justifica costo.

PostgreSQL on-premise: Mejor en analytics complejos pero tu caso es transaccional clásico (OLTP): inserts frecuentes, updates batch, joins simples por fecha/ID. MySQL cubre 100% sin overhead VACUUM PostgreSQL.

Conclusión: Tu patrón de uso (2K inserts/día + cruces + updates masivos) es caso de libro para bases transaccionales (MySQL/Azure SQL), NO para data warehouses analíticos (BigQuery). BigQuery brilla en read-only analytics pesados, pero tu proceso nocturno es write-heavy.
